# 사용자 인터뷰 — 쉬운 말로 안내하기

이 문서는 Claude가 사용자에게 질문할 때 **어떤 말투로, 어떤 설명과 예시를 곁들여** 물어봐야 하는지 규정한다.

## 핵심 원칙

1. **전문용어 금지** — 등장 시 즉시 괄호로 풀어쓰기
2. **예시 1개 이상** — 매 질문마다 실제 예시 제시
3. **왜 묻는지** — 배경·목적을 한 줄로 설명
4. **선택지 제공** — 가능한 답변을 나열 (자유입력은 최소화)
5. **"모르면 넘어가도 됨"** — 답변 부담 낮추기
6. **공포 유발 금지** — 과태료·처벌을 위협으로 쓰지 말 것

전문용어는 `references/glossary.md`를 참고해 풀어쓴다.

---

## 안내 템플릿

매 질문은 다음 4요소를 포함:

```
[왜 묻는지] 한 줄
[질문] 구체적 내용
[예시] 실제 답변 예시
[선택] 가능한 답변 or 자유입력 힌트
```

### 나쁜 예 (바로 잡아야 할 방식)
```
Q: CPO 정보를 입력하세요.
  - cpoName
  - cpoTitle
  - cpoPhone
  - cpoEmail
```

### 좋은 예
```
개인정보를 관리할 책임자를 정해야 해요.
이걸 '개인정보 보호책임자(CPO)'라고 부르는데, 법적으로 반드시 한 명 지정해야 합니다.

1인 창업이면 보통 대표님 본인이 되고, 회사 규모가 있으면 CTO나 법무팀장이 맡는 경우가 많아요.

다음 네 가지 알려주세요:
1. 이름 (예: 홍길동)
2. 직책 (예: 대표, CTO, 개인정보팀장)
3. 연락 가능한 전화번호 (예: 02-1234-5678)
4. 이메일 (예: privacy@회사도메인.kr)
```

## Step 0. 서비스 성격 스크리닝 (v2.1 재설계, 가장 먼저 묻는 질문)

**이전 버전의 "관할법/언어를 먼저 따로 묻는" 방식은 비효율이었음.** 이제 **세 가지만** 먼저 묻고, 그걸로 관할·언어·세부 질문을 자동 결정한다.

### 0-A. 서비스가 누굴 대상으로 해요?

```
안녕하세요. 먼저 서비스 성격부터 파악할게요.

어떤 사용자를 대상으로 하시나요?

1) 🇰🇷 한국 사용자만 (한국 내 서비스)
   예: 한국어 쇼핑몰, 국내 학원, 한국 앱

2) 🇪🇺 해외 사용자 위주 (EU·미국 등)
   예: 영어 B2B SaaS, 해외 마켓 대상 쇼핑몰

3) 🌍 양쪽 다 (한국+해외 글로벌 서비스)
   예: 한국 본사 + 유럽 진출, 다국어 지원 SaaS

4) 잘 모르겠어요 — 상담 시작
```

### 0-B. (3번 선택 시에만 추가) 해외 어느 지역이 주력?

```
글로벌 서비스네요. 해외 사용자가 어느 지역에 많나요?

1) 🇪🇺 유럽 (EU, 영국) — GDPR 적용
2) 🇺🇸 미국 — CCPA/CPRA (현재 로드맵 상태, EU 문서로 우선 대응 권장)
3) 🌏 아시아 (일본·중국·싱가포르 등) — 로드맵
4) 전 세계 고르게 — EU GDPR 기준으로 설계 (가장 엄격하므로 글로벌 대응 가능)
```

### 0-C. 운영 주체는 어느 나라에 있어요?

```
운영 주체(법인·사업자)가 어디에 있나요?

1) 🇰🇷 한국 법인·사업자
2) 🌏 해외 법인 (한국 사용자 있음) — 국내대리인 지정 필요
3) 해외 법인 (한국 사용자 없음)
```

### 자동 결정 로직

0-A, 0-B, 0-C 답을 종합해 다음 값을 **Claude가 결정하고 사용자에게 요약 확인**:

| 0-A | 0-B | 0-C | → `jurisdictions` | `outputLocale` |
|-----|-----|-----|------------------|---------------|
| 한국만 | - | 한국 | `["kr-pipa"]` | `ko` |
| 한국만 | - | 해외+한국 사용자 | `["kr-pipa"]` + 국내대리인 | `ko` |
| 해외 | EU | - | `["eu-gdpr"]` | `en` |
| 해외 | 미국 | - | `["eu-gdpr"]` (CCPA 대용) + 경고 | `en` |
| 양쪽 | EU | 한국 | `["kr-pipa", "eu-gdpr"]` | `both` |
| 양쪽 | 전 세계 | 한국 | `["kr-pipa", "eu-gdpr"]` | `both` |
| 모름 | - | - | 추가 상담 플로우 |

### 결정 요약 확인 단계

```
정리하면 이렇게 갑니다:

서비스 성격: {{serviceCharacter}}  (예: 한국 본사 + EU 사용자 대상 SaaS)
적용 법령: {{jurisdictions}}  (예: 한국 PIPA + EU GDPR)
출력 언어: {{outputLocale}}  (예: 한국어 + 영문 병기)
생성될 문서:
  · /privacy (한국어, PIPA 기준)
  · /terms (한국어)
  · /eu/privacy (영문, GDPR 기준)
  · /eu/terms (영문)
  · 회원가입 동의 모달 (한/영 자동 전환)
  · 쿠키 배너 (GDPR 옵트인 모드)

총 약 {{questionCount}}개 질문 드릴게요. 대략 10분 걸립니다.

이대로 진행해도 될까요? [예, 시작 / 바꾸고 싶어요]
```

### 이후 질문 동적 축소 규칙

`jurisdictions` 결정에 따라 **불필요한 질문 자동 제거**:

#### `["kr-pipa"]` 단독인 경우
- 묻는다: CPO, 사업자등록번호, 고충처리 부서
- 묻지 않는다: DPO, EU 대표자, GDPR 법적 근거, 국제 이전 SCCs

#### `["eu-gdpr"]` 단독인 경우
- 묻는다: Data Controller, DPO(선택), EU Representative, Legal Basis(Art. 6 근거 6종 중), Data Transfer Mechanism
- 묻지 않는다: 한국 CPO, 전자상거래법 보유기간, 14세 미만(EU는 16세), 국내대리인

#### `["kr-pipa", "eu-gdpr"]` 병기인 경우
- 양쪽 다 묻지만, 공통 항목은 **1회만** (서비스명, 수집 항목 등)
- 한국 전용: CPO, 사업자등록번호
- EU 전용: DPO, EU Representative, Art. 6 법적 근거, 국제 이전 메커니즘
- 아동: 한국 14세 / EU 16세 → 둘 다 처리 (더 엄격한 16세 기준으로 통일 권장)

## Step 1. 서비스 소개

### Q1-1. 서비스 이름
```
먼저 어떤 서비스를 운영하시나요?
서비스 이름을 알려주세요. (예: "모카샵", "우리학원 출결앱")
```

### Q1-2. 서비스 한 줄 설명
```
이 서비스가 한 줄로 어떤 건지 알려주세요.
처리방침·약관 서문에 들어갑니다.
예: "커피 원두 온라인 판매 플랫폼" / "초등학생 수학 학습 앱"
```

### Q1-3. 운영 주체
```
누가 이 서비스를 운영하시나요?
법인(회사)이면 정식 회사명, 개인사업자면 상호명, 개인 프로젝트면 본인 이름을 입력하시면 돼요.
예: "주식회사 모카" / "홍길동 (개인)"
```

### Q1-4. 도메인
```
서비스 주소(도메인)가 있나요?
예: "mocashop.kr"
아직 없으면 "없음"이라고 답하셔도 괜찮아요. 나중에 수정 가능합니다.
```

### Q1-5. 서비스 유형

```
어떤 종류의 서비스인가요? 법적으로 필요한 조항이 유형마다 달라서 확인이 필요해요.

1) SaaS / 웹앱 (구독형 소프트웨어)
   → 예: Notion, Figma, 사내 툴

2) 쇼핑몰 / 전자상거래 (물건·서비스 판매)
   → 예: 스마트스토어, 자사몰
   → 청약철회 같은 조항이 추가로 필요해요

3) 커뮤니티 / 소셜
   → 예: 게시판, 커뮤니티 앱
   → 게시물 저작권 조항이 들어갑니다

4) 블로그 / 개인 사이트
   → 예: 포트폴리오, 개인 블로그
   → 최소 필수 조항만 들어갑니다

5) 핀테크 / 금융 서비스
   → 예: 송금, 투자, 대출
   → 추가 법률 적용으로 조항이 많아요

6) AI 서비스
   → 예: AI 이력서 심사, AI 추천 알고리즘
   → '자동화된 결정' 관련 특별 조항이 추가됩니다
```

## Step 2. 어떤 정보를 모으나요

### Q2-1. 기본 수집 항목

```
회원가입이나 서비스 이용 시 고객한테서 어떤 정보를 받으세요?
해당되는 걸 모두 골라주세요.

A. 기본 정보
  [ ] 이름
  [ ] 이메일
  [ ] 전화번호
  [ ] 생년월일
  [ ] 성별

B. 서비스 유형별
  [ ] 주소 (배송지) — 쇼핑몰에서 주로
  [ ] 결제정보 (카드·계좌) — 유료 서비스
  [ ] 프로필 이미지
  [ ] 닉네임

C. 로그인 방식
  [ ] 자체 회원가입 (비밀번호 설정)
  [ ] Google 로그인
  [ ] Kakao 로그인
  [ ] Naver 로그인
  [ ] Apple 로그인

D. 자동 수집 (입력 안 받아도 쌓이는 정보)
  [ ] IP 주소, 쿠키, 기기정보
  → 웹/앱 서비스면 기본적으로 다 수집돼요. 체크 권장.

그리고 각 항목을 왜 받는지도 알려주세요.
예: "이메일은 로그인이랑 주문 확인 메일 보낼 때 써요"
```

### Q2-2. 민감정보 여부

```
혹시 '민감정보'를 받으세요? 민감정보란 법에서 특별히 조심스럽게 다뤄야 한다고 정한 정보예요.

- 건강 정보 (예: 운동앱에서 심박수, 의료앱에서 질환)
- 종교·사상·정치 성향
- 생체정보 (지문, 얼굴 인식)
- 범죄 경력
- 노동조합 가입 여부

일반적인 쇼핑몰·SaaS면 대부분 "해당 없음"이에요. 있으면 어떤 건지 알려주세요.
```

### Q2-3. 고유식별정보 여부

```
다음 네 가지 중 받는 게 있나요?

- 주민등록번호
- 여권번호
- 운전면허번호
- 외국인등록번호

주의: 이 정보들은 법에서 특별히 지정한 번호라, 받으려면 **법령 근거가 있어야** 합니다.
일반적으로 금융·통신·의료 서비스만 수집해요. 쇼핑몰·SaaS는 대부분 "해당 없음".

만약 받으신다면 어떤 법령 근거인지도 알려주세요.
예: "주민번호 — 전자금융거래법 제21조"
```

## Step 3. 수집하는 방법

```
고객 정보는 어떻게 받으세요? 해당되는 것 모두 선택:

  [ ] 서비스 이용 중 자동으로 쌓임 (로그·쿠키)
  [ ] 외부 제휴사가 넘겨줌
```

```

카테고리별로 골라주세요:

A. 회원 관리
  [ ] 회원 식별·본인 확인
  [ ] 부정 이용 방지
  [ ] 가입 의사 확인

B. 서비스 제공
  [ ] 주문·결제·배송
  [ ] 콘텐츠 제공
  [ ] 고객 문의 응대
  [ ] 맞춤형 추천

C. 개선·분석
  [ ] 서비스 이용 통계·분석
  [ ] 신규 서비스 개발
  [ ] 이벤트·설문

D. 마케팅 (별도 동의 필요)
  [ ] 광고·이벤트 안내
  [ ] 맞춤형 광고

각 카테고리를 고르셨으면 어떤 식으로 쓰는지 한 줄씩 설명해주세요.
```

## Step 5. 다른 회사에 정보를 넘기나요

```
우리 회사 밖으로 고객 정보를 **넘기는 경우**가 있나요?

'제3자 제공'이라고 하는데, 예를 들어:
- 쇼핑몰 → 택배회사에 배송지 전달
- 여행 앱 → 숙소에 예약자 정보 전달
- 제휴 마케팅 → 파트너사에 정보 공유

예/아니오로 답해주세요. 단순히 "AWS에 저장"이나 "Stripe로 결제" 같은 건 여기가 아니라 다음 단계(처리위탁)에서 다룹니다.
```

예라면 이어서:
```
어떤 회사에, 어떤 정보를, 왜 넘기는지, 그 회사는 얼마나 보관하는지 알려주세요.
예:
  - 대상: CJ대한통운
  - 항목: 이름, 주소, 전화번호
  - 목적: 상품 배송
  - 보유: 배송 완료 후 3개월
```

## Step 7. 책임자 정보

```
이제 중요한 정보예요. '개인정보 보호책임자'를 한 명 지정해야 합니다.

이게 뭐냐면: 고객 정보를 책임지고 관리하는 담당자예요.
법에서 반드시 한 명 지정해서 공개하라고 정해놨습니다.

1인 창업이시면 → 대표님 본인이 되시면 돼요
회사 규모가 있으면 → CTO, 법무팀장, 개인정보팀장 등이 맡습니다

다음 네 가지 알려주세요:

1. 이름 (예: 홍길동)
2. 직책 (예: 대표, CTO, 개인정보팀장)
3. 연락 가능한 전화번호 (예: 02-1234-5678)
   → 대표번호나 업무 휴대폰도 OK
4. 이메일 (예: privacy@회사도메인.kr)
   → privacy@ 나 cpo@ 같은 전용 주소를 많이 씁니다
```

그 다음:
```
고객 문의·불만을 응대할 담당 부서가 따로 있나요?
예: 고객센터, 지원팀이 별도로 있는 경우

없으시면 위 개인정보 보호책임자가 같이 맡으시면 돼요.
```

## Step 8. 특별한 경우 확인

한 번에 한 질문씩. 답변 "아니오"면 관련 섹션 생성 안 됨.

### Q8-1. 만 14세 미만 아동
```
만 14세 미만 어린이도 서비스를 쓸 수 있나요?

쓸 수 있다면 '법정대리인'(=부모님) 동의를 꼭 받아야 해서 추가 조항이 필요합니다.

일반 성인 대상 서비스면 → "아니오"
어린이 대상 학습앱·게임·SNS면 → "예"
```

예라면 이어서:
```
법정대리인 동의를 어떻게 확인하시나요?
- 부모님 휴대폰 본인인증
- 이메일 인증
- 신용카드 확인
- 기타
```

### Q8-2. AI 자동화 결정
```
AI나 알고리즘이 고객의 합격·불합격, 승인·거절 같은 걸 자동으로 결정하나요?

예시:
- 이력서를 AI가 자동 심사
- 대출 신청을 자동으로 승인/거절
- 추천 알고리즘으로 개인화 콘텐츠 노출

단순 통계 분석이나 검색 랭킹은 해당 안 돼요.
있으시면 "예", 없으면 "아니오".
```

예라면 이어서:
```
어떤 자동화 결정을 하시는지 설명해주세요.
- 어떤 서비스에서? (예: 이력서 AI 심사)
- 어떤 데이터를 기준으로? (예: 자기소개서, 경력)
- 어떤 식으로 평가? (예: 키워드 매칭 + 가중치)
```

### Q8-3. 해외 사업자
```
이 서비스 운영 법인이 **해외에 있나요**?
(한국에 본사·지사 없이 해외에서 한국 고객을 대상으로 서비스하는 경우)

예: 미국 법인이 한국 사용자에게 웹서비스 제공
→ 이 경우 '국내대리인'을 지정해야 해요

일반적인 한국 스타트업이면 "아니오"
```

예라면 이어서:
```
한국에 연락 창구 역할을 해줄 대리인 정보:
- 이름(법인명)
- 주소
- 전화번호
- 이메일
```

### Q8-4. 마케팅 수신
```
광고·이벤트 정보를 이메일·문자로 보내실 건가요?

보내시려면 회원가입 시 **별도 동의**를 받아야 합니다.
이 스킬이 동의 모달에 "마케팅 수신 동의" 항목을 선택 동의로 넣어드려요.
```

### Q8-5. 맞춤형 광고
```
Google Analytics, Meta Pixel 같은 **추적 도구**를 써서 맞춤형 광고를 하시나요?

웹사이트에 붙이는 픽셀 코드로 "이 사람 어떤 페이지 봤다"를 기록하고, 나중에 Facebook·Instagram에서 관련 광고를 보여주는 방식이에요.

쓰시면 "예" → 쿠키 배너 + 관련 조항 자동 추가
```

### Q8-6. 전송요구권
```
대부분 "아니오"이지만 일단 확인:

매출이 큰 대기업이나 플랫폼은 고객이 "내 정보 다른 서비스로 옮겨줘" 요청할 수 있는 권리를 보장해야 합니다.

일반 스타트업·중소기업이면 "아니오" 선택해주세요.
```

## Step 9.5. 디자인 스타일 (자동 감지 후 확인)

**이 단계는 Claude가 먼저 프로젝트를 분석한 후 시작한다.** `references/design-system-detection.md` 참조.

### 스킬의 기본 디자인 철학

법률 문서는 "신뢰감 있고 정제된" 톤이 기본. 사용자가 커스터마이즈하지 않으면 다음을 기본 적용:

- 폰트: **Pretendard Variable** (한글 모던 산세리프, CDN 방식 자동 설치)
- 색상: **흑백 모노크롬** (brand primary 대신 `#0a0a0a` 계열)
- 장식 최소: 이모지 금지, 볼드 최소화, 마크다운 헤딩 수준 감소
- 라벨링: 보더만 있는 격자형, 아이콘 없이 텍스트 중심

브랜드 감성 있는 페이지를 원하면 Step 9.5에서 명시적 선택 필요.

### 자동 감지 결과 제시

```
[프로젝트 디자인 분석 결과]

UI 라이브러리: shadcn/ui (감지됨)
아이콘 세트: lucide-react (감지됨, 기본은 아이콘 없음 미니멀)
모서리 스타일: --radius 0.5rem → '기본(default)'
다크모드: next-themes 있음 → 자동 대응
폰트: Pretendard Variable 자동 추가 예정
테마: 흑백 모노크롬 (법률 문서 기본 스타일)

이대로 만들어도 될까요? [이대로 / 바꾸고 싶어요]
```

### "바꾸고 싶어요" 선택 시에만 세부 질문

#### Q11-1. 모달 크기 & 스타일
```
회원가입 동의 모달을 어떻게 만들까요?

1) 기본 (사이즈 중간, 둥근 모서리) — 대부분 여기
2) 컴팩트 (작게, 미니멀)
3) 크게 (법률 내용 전부 노출)
4) 미니멀 (테두리 없고 깔끔)
```

#### Q11-2. 쿠키 배너 위치
```
쿠키 동의 배너를 어디에 띄울까요?

1) 하단 바 (기본, 사이트 전체 폭)
2) 우측 하단 플로팅 카드 (Google 스타일)
3) 상단 바
4) 중앙 모달 (첫 방문 시 차단형, GDPR 스타일)
```

#### Q11-3. 모서리 스타일
```
버튼·카드 모서리는 어느 정도 둥글게?

1) 감지된 --radius 값 사용 (권장, 프로젝트와 일관)
2) 사각 (모서리 없음)
3) 약간 둥글게
4) 많이 둥글게 (친근한 느낌)
```

#### Q11-4. 아이콘
```
아이콘 라이브러리:

1) 감지된 것 그대로 사용 (lucide-react)
2) 다른 라이브러리 사용 — 어떤 거요? (Heroicons/Tabler/Phosphor)
3) 아이콘 없이 텍스트만
```

#### Q11-5. 컬러 테마
```
주 컬러는 어떻게 할까요?

1) 흑백 모노크롬 (기본, 법률 문서 격에 맞음)
2) 프로젝트에서 감지된 --primary 색상 사용 (브랜드 감성 원할 때)
3) 중립(neutral) 회색 계열만
4) 직접 지정할게요 — 색상 HEX/CSS 변수 입력
```

#### Q11-6. 폰트
```
폰트는 어떻게 할까요?

1) Pretendard (기본, 한글 모던 산세리프, 자동 설치)
2) 프로젝트에서 이미 쓰는 폰트 그대로 사용
3) 다른 웹폰트 사용 - 이름 알려주세요
```

### 최종 variant 결과 예시

```yaml
styleVariant:
  modal: default       # default | compact | large | minimal
  cookieBanner: bottom-bar  # bottom-bar | floating | top-bar | center-modal
  radius: auto         # auto | none | sm | md | lg
  iconSet: lucide      # lucide | heroicons | tabler | phosphor | none
  colorScheme: primary # primary | neutral | custom(#hex)
```

이 값이 파일 생성 시 컴포넌트 props로 박힘.
